# 数据库设计：标准、规范与完整设计流程

数据库设计的核心目标：**数据结构清晰、无冗余、易维护、高性能、可扩展**，是软件开发的核心基石。我会用最通俗、工程化的方式给你讲清楚**标准/规范**、**设计步骤**、**实操规则**。

---

## 一、数据库设计的通用标准（行业公认准则）

这是所有设计必须遵守的**底层原则**：

1. **业务优先**：设计必须完全贴合业务需求，不脱离实际场景
2. **数据完整性**：保证数据准确、不重复、不缺失
3. **低冗余性**：相同数据只存一份，避免浪费空间+更新异常
4. **高性能**：查询快、写入稳，支持索引、分表等优化
5. **可扩展性**：方便后续加字段、加表、扩容
6. **安全性**：敏感数据加密、权限控制
7. **可读性**：命名统一、注释完整，别人能看懂

---

## 二、数据库设计规范（强制遵守的规则）

分为**命名规范**、**字段规范**、**表结构规范**、**索引规范**四大类。

### 1. 命名规范（统一风格，最基础）

- 全部使用**小写英文 + 下划线**，禁止中文、大小写混用、特殊符号
- 表名：使用**名词复数/业务含义**，如 `user`、`order`、`product_category`
- 字段名：见名知意，禁止 `a1/b2/c3` 无意义命名
- 关联字段：主表id = `id`，从表关联 = `表名_id`，如 `user_id`
- 禁止使用数据库关键字（`order`、`status` 等慎用）

### 2. 字段设计规范

- 必须有**主键**：`id`（自增/雪花ID）
- 必须有**公共字段**（行业标准）：
  - `id` 主键
  - `create_time` 创建时间
  - `update_time` 更新时间
  - `is_deleted` 软删除标记（0未删/1已删）
- 字段类型**按需选择**：
  - 数字用 int/bigint
  - 字符串用 varchar（限定长度）
  - 文本用 text
  - 时间用 datetime/timestamp
- 禁止允许 `null`的字段滥用：优先给**默认值**（0、''、'0000-00-00'）
- 字段长度合理，不浪费空间

### 3. 表结构规范

- 一张表只做**一件事**（用户表只存用户，订单表只存订单）
- 禁止大表无限扩张，超过20~30字段考虑拆表
- 关联关系清晰：一对多、多对多严格按范式设计
- 禁止冗余字段泛滥

### 4. 索引规范

- 主键自带索引
- **查询频繁的字段**加索引（where、order by、group by 字段）
- 禁止无意义索引（影响写入性能）
- 联合索引遵循**最左前缀原则**

## 三、数据库三大范式（设计核心规范）

范式 = 减少冗余、保证数据一致性的**标准化规则**
实际开发中**遵守前三范式即可**，更高范式不推荐（会影响性能）。

### 第一范式（1NF）

**列不可再分**，每一列都是原子数据
例：不能把“省市区+详细地址”存在一个字段里，应拆分。

### 第二范式（2NF）

满足1NF + **非主键字段完全依赖主键**，不能只依赖主键一部分
主要针对**联合主键**场景，禁止部分依赖。

### 第三范式（3NF）

满足2NF + **非主键字段之间不能互相依赖**
例：订单表里不要存“用户姓名、用户电话”，只存 `user_id`关联即可。

## 四、数据库完整设计步骤（从0到1）

这是企业真实开发的**标准流程**，你照着做就能设计出合格的库。

### 步骤1：需求分析（最关键）

梳理业务实体和关系，例如：

- 电商：用户、商品、订单、购物车、分类、评论
- 后台管理：用户、角色、权限、日志

输出：**实体清单 + 业务流程**

### 步骤2：概念结构设计（画E-R图）

E-R图 = 实体-关系图

- 矩形：实体=表
- 椭圆：属性=字段
- 菱形：关系=一对多/多对多

工具：Navicat、PowerDesigner、ProcessOn、Draw.io

### 步骤3：逻辑结构设计（转表结构）

把E-R图转换成具体表、字段、主键外键、关联关系。
遵守**三大范式**。

### 步骤4：物理结构设计（落地建表）

- 选择数据库（MySQL、PostgreSQL等）
- 定义字段类型、长度、索引、默认值、注释
- 设计分表、分区（大业务）

### 步骤5：优化与维护

- 加索引优化查询
- 反范式优化（适当冗余提升性能）
- 加公共字段、约束、唯一键
- 完善注释

## 五、企业级设计模板（直接套用）

### 标准表结构示例

```sql
CREATE TABLE `user` (
  `id` bigint NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `username` varchar(50) NOT NULL COMMENT '用户名',
  `password` varchar(100) NOT NULL COMMENT '密码（加密）',
  `phone` varchar(20) DEFAULT '' COMMENT '手机号',
  `status` tinyint NOT NULL DEFAULT '1' COMMENT '状态 1正常 0禁用',
  `create_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `is_deleted` tinyint NOT NULL DEFAULT '0' COMMENT '是否删除 0否 1是',
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';
```

### 必须遵守的“三行铁律”

1. **每张表必须有 id、create_time、update_time、is_deleted**
2. **命名统一小写+下划线**
3. **不存储可推导数据，不冗余存储**

---

## 六、重要提醒：反范式设计

范式是为了规范，但**过度范式会导致查询慢**。
实际开发中：

- 读多写少 → 允许**适当冗余**（如订单存用户名）
- 写多读少 → 严格遵守范式

---

### 总结

1. **标准**：业务贴合、低冗余、高性能、可扩展
2. **规范**：命名规范、字段规范、表规范、索引规范 + 三大范式
3. **设计流程**：需求分析 → E-R图 → 逻辑设计 → 物理设计 → 优化
4. **实操模板**：每张表必备 id、时间、软删除字段
