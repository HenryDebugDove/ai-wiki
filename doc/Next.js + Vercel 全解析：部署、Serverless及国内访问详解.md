# Next.js + Vercel 全解析：部署、Serverless 及国内访问详解

在前端全栈开发中，Next.js 和 Vercel 是黄金搭档，很多开发者从纯前端项目入手，逐步拓展到后端接口，过程中会遇到部署、Serverless 使用、国内访问等一系列问题。本文结合实际开发场景，把这些核心问题讲透，全程用通俗表述，搭配表格对比，方便快速理解和实操。

## 一、Next.js 优缺点详解

Next.js 是基于 React 的全栈框架，既能做前端页面，也能写后端接口，是目前最流行的前端框架之一，其优缺点非常鲜明，适合不同场景的项目。

| 类别 | 具体内容 | 补充说明 |
|------|----------|----------|
| 优点 | 自带路由，无需额外配置 | 按文件目录自动生成路由，不用写路由配置代码，比如 app/page.js 就是首页，app/api/xxx 就是接口路由 |
| 优点 | 支持多种渲染模式 | 可实现 SSR（服务端渲染）、SSG（静态生成）、ISR（增量静态再生），兼顾 SEO 和页面加载速度，适合博客、官网等需要 SEO 的场景 |
| 优点 | 原生支持后端接口 | 不用额外搭建后端框架，在 app/api 目录下写代码，就是后端接口，部署后自动转为 Serverless 函数 |
| 优点 | 开发体验好，热更新快 | 本地运行 npm run dev 后，修改代码实时生效，不用手动刷新页面，节省开发时间 |
| 优点 | TypeScript 友好 | 官方原生支持 TypeScript，不用额外配置，能减少代码 bug，提升项目可维护性，尤其适合团队协作和大型项目 |
| 优点 | 无缝对接 Vercel，部署简单 | 和 Vercel 同属一个团队开发，零配置部署，推代码到 GitHub 就能自动上线 |
| 缺点 | 学习成本略高 | 如果是新手，需要同时掌握 React 和 Next.js 的路由、渲染模式等特性，比纯 React 入门稍难，尤其是 SSR/ISR 概念需要时间理解 |
| 缺点 | 打包体积较大 | 默认打包会包含很多框架相关代码，纯静态项目比传统 HTML/CSS/JS 体积大，需要手动优化 |
| 缺点 | 部分配置不够灵活 | Next.js 封装了很多底层配置，比如 webpack 配置，想做深度自定义需要额外学习配置方法，自由度不如纯 React 项目 |
| 缺点 | 静态导出有局限性 | 如果需要静态导出部署（比如部署到 GitHub Pages），会限制部分 Next.js 特性，比如图片优化需要手动关闭 |

## 二、GitHub 快速部署 Next.js 到 Vercel（全程零配置）

Vercel 是 Next.js 官方推荐的部署平台，部署过程非常简单，只要项目已经上传到 GitHub，全程点击鼠标就能完成，不用写任何部署脚本。

### 部署前提

- 项目已上传到 GitHub（确保包含 package.json、next.config.js 等核心文件，.next 文件夹不用上传，.gitignore 会自动忽略）
- 注册并登录 Vercel（推荐用 GitHub 账号直接登录，后续无需额外授权）

### 部署步骤（4 步搞定）

| 步骤 | 具体操作 | 注意事项 |
|------|----------|----------|
| 1 | 登录 Vercel，点击右上角「New Project」，选择「Import from Git」 | 确保 Vercel 已授权访问你的 GitHub 账号，能看到你的项目列表 |
| 2 | 选择需要部署的 Next.js 项目，点击「Import」 | 不用修改任何配置，Vercel 会自动识别 Next.js 框架，自动填充构建命令 |
| 3 | 直接点击「Deploy」，等待 1-2 分钟，部署自动完成 | 部署过程中会显示构建日志，若报错，大概率是依赖缺失或环境变量未配置 |
| 4 | 部署完成后，Vercel 会生成一个免费域名（xxx.vercel.app），点击即可访问线上项目 | 后续修改代码，只要 git push 到 GitHub，Vercel 会自动重新部署，不用手动操作 |

### 部署后关键说明

1. **自动部署**：绑定 GitHub 后，每次 push 代码到 main 分支，Vercel 会自动拉取代码、构建、部署，全程零手动操作；
2. **环境变量**：如果项目用到 .env 文件中的密钥（比如数据库密码、API Key），需要在 Vercel 项目 → Settings → Environment Variables 中手动添加，否则接口会报错；
3. **预览环境**：提 PR 时，Vercel 会自动生成预览链接，方便测试，不会影响线上主站。

## 三、Vercel 支持部署的项目类型（不止 Next.js）

很多人以为 Vercel 只能部署 Next.js 项目，其实它支持绝大多数现代前端、全栈项目，甚至 Serverless 形态的后端项目，只是对 Next.js 支持最完善（官方原生）。

| 项目类型 | 具体框架/技术 | 支持度 | 说明 |
|----------|---------------|--------|------|
| 全栈框架（推荐） | Next.js | ⭐⭐⭐⭐⭐ | 官方原生支持，所有特性（SSR/ISR/API Routes）都能完美适配，部署零配置，是 Vercel 最推荐的项目类型 |
| 全栈框架（推荐） | Nuxt.js（Vue 全栈）、SvelteKit（Svelte 全栈）、Remix | ⭐⭐⭐⭐⭐ | 和 Next.js 类似，自带路由、后端接口，Vercel 能自动识别，一键部署，其中 SvelteKit 是 Svelte 官方推荐部署到 Vercel |
| 纯前端框架 | React（Create React App）、Vue、Angular | ⭐⭐⭐⭐ | 自动识别构建命令，无需额外配置，部署后自动开启 CDN 加速，适合纯静态 SPA 项目 |
| 纯前端框架 | Vite + React/Vue | ⭐⭐⭐⭐⭐ | 快速识别 Vite 项目，自动填充构建命令（vite build）和输出目录（dist），部署速度快 |
| 静态站点 | HTML/CSS/JS 纯静态、Hexo、VuePress、Hugo、Astro | ⭐⭐⭐⭐⭐ | 直接上传就能部署，自动开启静态资源缓存，适合博客、文档站、落地页等纯静态项目 |
| Serverless 后端 | Node.js（Express、NestJS、Elysia） | ⭐⭐⭐⭐ | 需适配 Serverless 形态，通过 vercel.json 配置路由转发，不支持长驻进程，适合轻量 API 服务 |
| Serverless 后端 | Python（FastAPI、Flask） | ⭐⭐⭐⭐ | 在 api 目录下编写代码，Vercel 自动识别 Python 运行时，部署后直接访问接口 |
| Serverless 后端 | Go、Rust | ⭐⭐⭐ | 编译为二进制文件，通过 vercel.json 配置为 Serverless 函数，适合轻量、高性能的 API 服务 |
| Edge Functions（边缘函数） | - | ⭐⭐⭐⭐⭐ | 全球边缘节点运行，适合做鉴权、路由重写、A/B 测试，Next.js Middleware 自动转为 Edge 函数 |
| 不推荐部署 | Java SpringBoot、PHP-FPM、容器化应用、长驻服务 | ⭐ | Vercel 基于 Serverless 架构，不支持长驻进程，这类项目适合部署到阿里云 ECS、Render 等平台 |

## 四、Next.js 自带的 Serverless 详解

很多开发者不知道，Next.js 自带的后端接口，部署到 Vercel 后，全部自动变成 Serverless 形态，不用手动配置，这也是 Next.js 做全栈开发的核心优势。

### 1. 什么是 Next.js 的 Serverless？

简单说，Next.js 的 Serverless 就是：你在 app/api 目录下写的后端接口，部署到 Vercel 后，会被自动拆成一个个独立的“函数”，有人访问时才执行，执行完就销毁，不用你维护服务器，也不用开机、重启。

你不用懂 Serverless 的底层原理，只要按照 Next.js 的规范写接口，部署后就自动是 Serverless 架构。

### 2. Next.js 中哪些功能是 Serverless？

| 功能类型 | 具体写法 | 说明 |
|----------|----------|------|
| App Router 接口（常用） | app/api/xxx/route.js（或 .ts） | 最常用的后端接口写法，支持 GET、POST、PUT、DELETE 等请求方式，部署后自动转为 Serverless 函数 |
| Pages Router 接口（老写法） | pages/api/xxx.js | Next.js 旧版本的接口写法，现在仍支持，部署后同样是 Serverless 函数 |
| 服务端组件 | app/page.js（默认就是服务端组件） | 可以直接在组件中连数据库、调用第三方 API，代码不暴露给前端，本质也是 Serverless 形态 |
| Middleware 中间件 | middleware.js | 用于鉴权、路由重写等，部署后转为 Edge Serverless 函数，全球边缘节点运行，速度更快 |

### 3. Next.js Serverless 的核心特点

- **零配置**：不用手动开启 Serverless，Vercel 自动处理，写好接口 push 代码就部署；
- **自动扩容**：1 人访问和 10 万人访问一样稳，Vercel 自动扩缩容，不用管服务器压力；
- **免费够用**：个人项目免费额度（每月 100 万次请求、100 小时运行时间）完全够用，不用花钱；
- **有执行限制**：Vercel 免费版接口最长运行 10 秒，不能做长耗时操作（比如视频转码、大数据处理）。

## 五、Serverless 优缺点详细解析

Serverless 是一种“无服务器”的后端架构，核心是“按需执行、用完即销毁”，不仅 Next.js 能用，其他语言（Node.js、Python 等）也能部署为 Serverless 函数，其优缺点非常突出，适合特定场景。

| 类别 | 具体内容 | 通俗说明 |
|------|----------|----------|
| 优点 | 无需维护服务器 | 不用买服务器、不用配置 Nginx、PM2，不用开机、重启、打补丁，省去所有运维工作 |
| 优点 | 自动无限扩容 | 访问量突然暴涨（比如从 1 人到 10 万人），平台自动增加资源，不会卡顿、崩溃，访问量下降后自动缩减资源 |
| 优点 | 成本极低，按需付费 | 只有有人访问接口时才计费，免费额度基本够个人/小项目用，不用不花钱，比买服务器便宜太多 |
| 优点 | 部署简单，和前端同步 | 和前端代码放在同一个项目，push 代码就一起部署，不用分开部署前端和后端，节省时间 |
| 优点 | 高可用、高稳定 | 由 Vercel、阿里云等大平台兜底，服务器集群分布在全球，几乎不会出现整体宕机的情况 |
| 优点 | 安全省心 | 平台会自动防护 DDoS 攻击、修复系统漏洞，不用自己做安全配置，降低安全风险 |
| 优点 | 开发速度快 | 专注写业务逻辑（比如接口、数据处理），不用花时间搞运维，适合快速开发、快速迭代项目（比如 MVP 原型） |
| 缺点 | 不能长驻运行 | 函数执行完就销毁，不支持 WebSocket、长连接、定时循环，比如聊天系统、实时通知就不适合用 Serverless |
| 缺点 | 执行时间有限制 | Vercel 免费版最长 10 秒，Pro 版最长 60 秒，不能做长耗时操作（比如视频转码、爬几十页数据） |
| 缺点 | 冷启动延迟 | 一段时间没人访问，第一次访问会慢几百毫秒（冷启动），后续访问会变快，对体验有轻微影响 |
| 缺点 | 没有本地文件存储 | 不能往服务器本地写文件（比如用 fs 模块写数据），文件必须存到云存储（Vercel Blob、阿里云 OSS 等） |
| 缺点 | 调试麻烦 | 本地环境和线上 Serverless 环境有差异，日志不如传统服务器直观，排查问题比传统后端麻烦 |
| 缺点 | 不支持随意安装系统依赖 | 比如 Chrome、FFmpeg 等系统级工具，不一定能安装，限制了部分功能的实现 |
| 缺点 | 平台绑定（vendor lock-in） | 在 Vercel 上写的 Serverless 函数，不能直接迁移到阿里云、腾讯云，需要修改代码适配，迁移成本高 |

### Serverless 适用与不适用场景

| 适用场景 | 不适用场景 |
|----------|------------|
| 1. Next.js 全栈项目的 API 接口（增删改查、登录注册）；2. 表单提交、数据收集（留言、问卷）；3. 第三方 API 代理（防跨域、隐藏密钥）；4. 支付回调、Webhook 接收；5. 图片压缩、验证码发送等轻量处理；6. 轻量定时任务（每日统计、报表推送）；7. 短链接服务、跳转服务 | 1. 聊天系统、游戏服务器（需要 WebSocket、长连接）；2. 视频转码、大数据处理（长耗时操作）；3. 传统后端服务（Java SpringBoot、PHP 长驻服务）；4. 需要本地文件存储的项目；5. 高并发、长耗时的企业级后端服务 |

## 六、Vercel 部署国内无法访问的问题（解决方案）

很多开发者部署完 Next.js 项目后，发现国内用户打不开，核心原因是 Vercel 自带的 xxx.vercel.app 域名被国内 DNS 污染/屏蔽，不是项目本身的问题，只要做好配置，就能让国内用户正常访问。

### 1. 国内无法访问的核心原因

| 访问方式 | 国内访问情况 | 原因 |
|----------|--------------|------|
| Vercel 自带域名（xxx.vercel.app） | 基本无法访问，需科学上网 | *.vercel.app 后缀域名被国内 DNS 污染，解析时会返回错误 IP，导致无法连接 |
| 绑定自定义域名（xxx.com） | 可正常访问，但速度一般、偶尔波动 | 自定义域名未被屏蔽，通过正确解析，可绕开 DNS 污染，但 Vercel 服务器在海外，跨境访问有延迟 |

### 2. 解决方案（3 步搞定，国内可直连）

核心思路：绑定自己的自定义域名 + 正确 DNS 解析，无需备案（服务器在海外），就能让国内用户正常访问，若想提升速度，可搭配 Cloudflare CDN 加速。

| 步骤 | 具体操作 | 注意事项 |
|------|----------|----------|
| 1. 购买自定义域名 | 在阿里云、腾讯云等平台购买域名（比如 todo.com、myapp.cn），每年费用约 30 元，无需备案 | 选择常见后缀（.com、.cn），避免小众后缀，解析更稳定 |
| 2. Vercel 后台绑定域名 | 1. 进入 Vercel 项目 → Settings → Domains；2. 输入你的自定义域名（比如 www.todo.com）；3. Vercel 会自动生成 DNS 解析记录，优先选择国内优化的 CNAME 记录（cname-china.vercel-dns.com） | 不要选错解析记录，国内用户优先用 cname-china.vercel-dns.com，解析速度更快 |
| 3. 域名服务商配置 DNS 解析 | 1. 登录域名服务商（阿里云/腾讯云）控制台 → DNS 解析；2. 添加 CNAME 记录：主机记录填 www（或 @），记录值填 Vercel 生成的 CNAME 地址；3. 保存后，等待 10 分钟左右（DNS 解析生效时间） | 记录类型必须选 CNAME，不要选 A 记录；生效后，访问自定义域名即可正常打开项目 |

### 3. 进阶优化：提升国内访问速度

如果绑定自定义域名后，国内访问速度较慢，可搭配 Cloudflare CDN 加速（免费版即可），步骤如下：

1. 登录 Cloudflare，添加你的自定义域名，按照提示修改域名的 DNS 服务器为 Cloudflare 提供的地址；
2. 在 Cloudflare DNS 页面，添加 CNAME 解析，目标填 Vercel 生成的地址，开启代理模式（橙色云朵图标，关键）；
3. 进入 Cloudflare 速度设置，开启自动压缩（HTML/CSS/JS）和智能路由，利用其国内节点提升访问速度。

### 4. 常见问题排查

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 绑定域名后，国内仍无法访问 | DNS 解析未生效，或解析记录填写错误 | 等待 30 分钟后重试，检查解析记录类型（CNAME）和记录值是否正确 |
| 访问速度慢、偶尔超时 | Vercel 服务器在海外，跨境访问延迟高 | 搭配 Cloudflare CDN 加速，或分离静态资源到阿里云 OSS/腾讯云 COS |
| 提示域名未备案，无法访问 | 误将域名解析到国内服务器，Vercel 服务器在海外，无需备案 | 检查解析记录，确保指向 Vercel 提供的 CNAME 地址，而非国内服务器 IP |

## 七、总结

Next.js 是一款功能强大的全栈框架，自带后端接口能力，部署到 Vercel 后自动转为 Serverless 形态，零配置、低成本，非常适合个人项目、学习 demo 和小型全栈项目；Vercel 不仅支持 Next.js，还能部署多种前端、全栈和 Serverless 项目，只是国内访问需要绑定自定义域名解决 DNS 污染问题。

Serverless 架构的核心是“按需执行、无需运维”，优点突出但也有明显限制，适合轻量、短耗时的后端任务；如果你的项目以国内用户为主，且需要更高的稳定性和访问速度，后期可迁移到阿里云 ECS 等国内平台，搭配 GitHub Actions 实现自动部署，兼顾开发体验和用户体验。
