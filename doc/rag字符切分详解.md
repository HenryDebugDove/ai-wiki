在RAG系统中，文本切分（Chunking）是非常关键的一步。 `LangChain RecursiveCharacterTextSplitter`，正是目前**最主流、最通用**的方案。


---

## 一、主流方案分层

| 级别             | 方案名称               | 核心逻辑                                                           | 适用场景                     |
| :--------------- | :--------------------- | :----------------------------------------------------------------- | :--------------------------- |
| **基础**   | 固定长度切分           | 按字符数/Token数强行截断                                           | 极简单场景，不推荐单独使用   |
| **主流⭐** | **递归字符切分** | 按**段落 → 句子 → 字符**的优先级递归切分，尽量保持语义完整 | **通用场景，最推荐**   |
| **进阶**   | 按文档结构切分         | 按Markdown标题、HTML标签、代码函数等结构切分                       | 结构化文档（技术文档、博客） |
| **进阶**   | 语义切分               | 用Embedding模型判断句子间的相似度，差异大处切分                    | 高精度场景，但计算成本高     |
| **专用**   | 按特殊边界切分         | 按Python函数、Java类等代码结构切分                                 | 代码库RAG                    |

---

## 二、重点展开：递归字符切分（最主流）

### 为什么它是主流？

- **简单有效**：不需要模型参与，纯规则，速度快
- **保持语义**：优先按段落切，切不了再按句子，实在不行才按字符
- **有重叠**：支持chunk_overlap，让相邻块共享部分内容，避免关键信息被切在边界上

### 工作原理示例

```
原始文本：段落1（300字） + 段落2（200字）+ 段落3（150字）

目标chunk_size = 250字

切分结果：
Chunk1: 段落1（250字，截断）+ chunk_overlap
Chunk2: 段落1剩余部分 + 段落2（组合到250字）
Chunk3: 段落2剩余部分 + 段落3（组合到250字）
Chunk4: 段落3剩余部分
```

### 代码示例（你项目中的用法）

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,        # 每个块最大500字符
    chunk_overlap=50,      # 相邻块重叠50字符
    separators=["\n\n", "\n", "。", "，", " "]  # 递归切分的优先级
)
chunks = splitter.split_text(document)
```

## 三、各方案对比总结

| 方案                   | 优点                         | 缺点                          | 推荐度               |
| :--------------------- | :--------------------------- | :---------------------------- | :------------------- |
| 固定长度切分           | 极简单，速度快               | 切断句子/段落，语义破坏严重   | ❌ 不推荐            |
| **递归字符切分** | 平衡性好，语义保留佳，速度快 | 无法理解深层语义              | ⭐⭐⭐⭐⭐           |
| 按文档结构切分         | 非常符合文档逻辑（如按标题） | 依赖文档格式规范              | ⭐⭐⭐⭐             |
| 语义切分               | 语义边界最准                 | 慢，成本高，需要Embedding模型 | ⭐⭐⭐（高精度场景） |
| 代码切分               | 保持函数/类完整性            | 仅限代码                      | ⭐⭐⭐               |

## 四、面试时的回答建议

如果面试官问“你用了什么切分方案？为什么？”

> “我在项目中主要用了**LangChain的递归字符切分器**，这是目前最主流的方案。它的核心思路是按**段落 → 句子 → 字符**的优先级递归切分，配合50字符的重叠，能在速度和语义完整性之间取得很好的平衡。
>
> 同时我也做了优化：针对技术文档，我会增加‘###’等Markdown标题作为优先切分符；对于代码块，我会保持其完整性不切碎。另外也实验了语义切分，但因为计算成本较高，目前只在高质量文档场景下开启。”
