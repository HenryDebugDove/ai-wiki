# 分布式锁核心问题总结（含结构图+表格+案例）

本文整合所有核心问题，包含结构图、对比表格、真实业务案例，全面拆解分布式锁的定义、适用场景、实现方式及Redis部署细节，兼顾理解和面试实用性。

# 一、分布式锁是什么 + 通俗例子（含对比）

## 1. 核心定义（对比本地锁）

| 锁类型                                                                                                                                               | 作用范围                      | 核心原理                                                                    | 适用场景                                     |
| ---------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------- | --------------------------------------------------------------------------- | -------------------------------------------- |
| ---------------------------------------------------------------------------------------------------------------------------------------------------- |                               |                                                                             |                                              |
| 本地锁（synchronized/ReentrantLock）                                                                                                                 | 单台服务器、单个JVM           | 基于JVM内存，锁住当前进程内的代码块，仅限制本机并发                         | 单机单实例项目，无多节点部署需求             |
| -                                                                                                                                                    | -                             | -                                                                           | -                                            |
| 分布式锁                                                                                                                                             | 多台服务器、多服务实例、跨JVM | 将锁存储在公共中间件（Redis/ZK/数据库），所有服务实例共同访问，实现全局互斥 | 多实例集群、负载均衡场景，多节点操作共享资源 |

## 2. 通俗类比例子（生活化+业务化）

### 例子1：生活化类比（公共卫生间）

* 单机单实例（无分布式锁）：1个厕所、1个入口，门口保安（本地锁）管控，进去一个锁门，无问题；
* 多实例+负载均衡（无分布式锁）：商场10个厕所、10个入口（10台服务实例），每个入口有自己的小锁（本地锁），大家会同时冲进同一个坑位，混乱不堪；
* 多实例+分布式锁：厕所大门口放1把唯一大锁（分布式锁，存在Redis），谁拿到钥匙谁进，不管多少入口，同一时间只允许1人使用，实现全局互斥。

### 例子2：真实业务案例（商品超卖）

场景：电商秒杀，商品库存仅剩5件，后端部署3台服务实例，前端通过Nginx负载均衡分发请求。

* 不加锁：3个用户同时下单，请求分别打到3台实例，3台实例同时查询库存（均显示5件），同时扣减库存，最终库存变成-2，出现超卖，数据错乱；
* 加分布式锁：以商品ID为锁Key（如lock:goods:10086），3台实例同时去Redis抢这把锁，只有1台实例抢到锁，执行“查库存→扣库存→生成订单”，执行完释放锁，下一个实例再抢锁，彻底杜绝超卖。

# 二、分布式锁是不是主要针对多实例、负载均衡场景

**结论：是，分布式锁的核心诞生场景就是多实例集群+负载均衡，没有这个场景，基本用不到分布式锁。**

## 1. 核心逻辑拆解

* 单机单实例：所有请求都在同一个JVM内存中，本地锁（synchronized）就能锁住代码，控制并发，无需分布式锁；
* 多实例+负载均衡：多台服务实例各自独立，内存隔离，本地锁只能锁住自己实例内的代码，其他实例完全感知不到，本地锁彻底失效；
* 分布式锁的作用：打破多实例的内存隔离，通过公共中间件（Redis等），让所有实例共用一把锁，实现跨机器、跨JVM的全局互斥，保护共享资源。

## 2. 部署结构图（一看就懂）

多实例+负载均衡+分布式锁的完整架构：

```plain
用户请求
       ↓
  Nginx/网关（负载均衡）  ——  分发请求到不同业务实例
       ↓
———————————————————  多实例业务集群（内存隔离，本地锁失效）
|  业务实例A  |  业务实例B  |  业务实例C  |
———————————————————
       ↓         ↓         ↓
       ————————————————————
               ↓
      【统一的Redis服务】  ——  分布式锁存储在这里（全局共享）
               ↓
      共享资源（数据库/缓存）
```

关键点：所有业务实例都连接同一个Redis，抢同一把锁，实现全局互斥，保护共享资源（库存、订单等）。

## 3. 补充：除了负载均衡，这些场景也需要分布式锁

本质都是“多节点操作同一份共享资源”，和负载均衡同源：

| 场景                                                                     | 不加分布式锁的问题                   |
| ------------------------------------------------------------------------ | ------------------------------------ |
| ------------------------------------------------------------------------ |                                      |
| 集群定时任务（每日报表、对账）                                           | 多台实例同时执行，重复统计、数据错乱 |
| -                                                                        | -                                    |
| 微服务跨服务操作（订单+库存）                                            | 多服务同时操作同一数据，出现脏写     |
| MQ消息重复消费                                                           | 同一条消息被多实例消费，重复更新状态 |

## 补充：企业实际选型建议

* 互联网高并发场景（秒杀、电商）：首选  **Redis分布式锁（Redisson框架）** ，无需手写锁逻辑，解决所有痛点；
* 金融、支付等强一致场景：选 **Zookeeper分布式锁** ，牺牲性能换数据安全；
* 小项目、内部后台（低并发）：可临时用  **数据库分布式锁** ，不推荐长期使用。

# 四、多实例业务下，Redis是不是只有一个（含部署架构）

核心结论： **业务多实例 ≠ Redis多实例** ，所有业务实例共用一套统一的Redis（单节点/集群），Redis多实例是为了高可用，不是每个业务实例配一个Redis。

## 1. 两个“多实例”的区别（关键区分）

| 类型                                                                                                                                                                                             | 核心目的                                               | 部署方式                                                                                 | 和分布式锁的关系                                                                       |
| ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------ | ---------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |                                                        |                                                                                          |                                                                                        |
| 业务多实例                                                                                                                                                                                       | 扛住用户请求、实现负载均衡，提高业务吞吐量             | 多台服务器部署，每台一个业务实例，前端通过网关负载均衡分发请求                           | 业务多实例是分布式锁的“使用场景”，没有业务多实例，就不需要分布式锁                   |
| -                                                                                                                                                                                                | -                                                      | -                                                                                        | -                                                                                      |
| Redis多实例（集群/主从）                                                                                                                                                                         | 保证Redis服务高可用，防止Redis单点宕机，避免锁服务失效 | 1. 基础版：单节点Redis（测试/小流量）；2. 生产版：Redis主从+哨兵/Redis集群（多节点部署） | Redis多实例是分布式锁的“载体保障”，确保锁服务不崩溃，所有业务实例共用这一套Redis集群 |

## 2. Redis部署结构图（生产环境常用）

```plain
———————————————————  业务多实例集群（负载均衡）
|  业务实例A  |  业务实例B  |  业务实例C  |
———————————————————
       ↓         ↓         ↓
       ————————————————————
               ↓
      【Redis集群（多实例）】  ——  全局共享，存储分布式锁
               ↓
      （主节点：处理锁请求；从节点：备份；哨兵：监控故障转移）
               ↓
      共享资源（数据库/缓存）
```

## 3. 关键补充说明

* 不是“每个业务实例配一个Redis”：如果每个业务实例单独配Redis，锁会分散在不同Redis中，无法实现全局互斥，分布式锁就失去意义；
* Redis单节点vs集群：测试环境可以用单节点Redis（简单），生产环境必须用Redis多实例（主从+哨兵/集群），防止Redis单点宕机，导致锁服务崩溃，影响业务；
* 核心逻辑：不管Redis是单节点还是多实例集群， **所有业务实例都连接同一套Redis** ，抢同一把锁，实现全局互斥。

# 五、整体总结

1. 分布式锁：跨多实例、跨JVM的全局锁，存储在公共中间件，解决多节点操作共享资源的并发问题，区别于仅能控制单机的本地锁；
2. 适用场景：核心是多实例集群+负载均衡，只要多节点操作同一共享资源（库存、订单等），就必须用分布式锁；单机单实例无需分布式锁；
3. 常见类型：Redis锁（主流，高并发）、Zookeeper锁（强一致，低并发）、数据库锁（临时方案），企业首选Redisson实现的Redis锁；
4. Redis部署：业务多实例和Redis多实例是两套独立部署，所有业务实例共用一套Redis（单节点/集群），Redis多实例是为了保证锁服务高可用，不是每个业务实例配一个Redis。
