# 前端模块化方案全面总结

## 核心重点：CommonJS + ES6 Module

前端模块化核心作用： **解决代码依赖混乱、全局变量污染、文件加载顺序问题** ，实现代码复用、解耦与高效维护。其中， **CommonJS** （服务端主流）和 **ES6 Module** （官方标准、前端主流）是当前最核心、最常用的两大方案，以下重点详解，其他方案仅作简要说明。

## 一、核心模块化方案对比表

| 规范名称 | 核心语法 | 加载方式 | 运行环境 | 业务使用场景 | 热门程度 | 核心特点 |
|---------|---------|---------|---------|-------------|---------|---------|
| **ES6 Module（ESM）** | import / export（default） | 编译时静态加载 | 现代浏览器+Node14+ | 所有现代前端项目（React/Vue/Vite）、官方标准场景 | ⭐⭐⭐⭐⭐（最高） | 官方标准、支持Tree-shaking、未来主流 |
| **CommonJS（CJS）** | require() / module.exports | 运行时同步加载 | Node.js 服务端为主 | Node后端开发、Webpack构建脚本、本地工具包 | ⭐⭐⭐⭐⭐（最高） | 服务端首选、语法简洁、不适合浏览器原生加载 |
| UMD | 兼容CJS+AMD+全局变量 | 自适应 | 全环境（浏览器+Node） | 通用SDK/工具库、跨环境兼容包 | ⭐⭐⭐ | 兼容万金油，过渡方案，非核心首选 |
| AMD | define() / require() | 异步加载 | 浏览器端 | 旧版浏览器老项目维护 | ⭐⭐ | 已淘汰，仅用于老项目兼容 |
| CMD | define(function(require, exports, module){}) | 异步加载 | 浏览器端 | 旧版Sea.js项目 | ⭐ | 基本淘汰，极少使用 |
| SystemJS | System.import() | 运行时动态加载 | 浏览器端 | 微前端、动态加载模块小众场景 | ⭐⭐⭐ | 小众专用，非通用模块化方案 |

## 二、ES6 Module 与 CommonJS 核心特点对比表

| 特性 | ES6 Module（ESM） | CommonJS（CJS） |
|------|------------------|----------------|
| 加载时机 | 编译时静态加载 | 运行时同步加载 |
| 加载方式 | 静态分析，支持Tree-shaking | 动态执行，无Tree-shaking |
| 模块引用 | 只读引用，双向绑定 | 值拷贝，单向传递 |
| 导出方式 | 命名导出 + 默认导出 | module.exports / exports |
| 导入方式 | import 语句 | require() 函数 |
| 浏览器支持 | 现代浏览器原生支持 | 需要打包工具转换 |
| Node.js支持 | Node 14+ 支持 | 所有Node.js版本默认支持 |
| 适用场景 | 现代前端项目、跨端项目 | Node.js服务端项目、构建脚本 |
| 语法特点 | 简洁、官方标准 | 直观、历史悠久 |
| 性能优化 | 支持Tree-shaking，打包体积小 | 无Tree-shaking，打包体积较大 |

## 三、重点模块方案详解

### 1. ES6 Module（ESM）—— 官方标准，现代前端首选

ES6 Module是ECMAScript官方推出的模块化规范，也是目前 **现代前端项目的唯一推荐方案** ，彻底解决了前端模块化的历史遗留问题，成为行业主流。

#### 核心语法

```javascript
// 1. 导出（两种常用方式）
// 方式1：命名导出（可多个）
export const name = "ESM模块";
export function add(a, b) {
  return a + b;
}

// 方式2：默认导出（仅一个）
export default {
  name: "默认导出模块",
  fn: () => console.log("默认导出")
};

// 2. 导入（对应导出方式）
// 导入命名导出的内容
import { name, add } from './module.js';

// 导入默认导出的内容（可自定义名称）
import myModule from './module.js';

// 3. 整体导入（命名导出批量导入）
import * as allModule from './module.js';
```

#### 核心特点

- **静态加载**：编译时解析依赖，而非运行时，支持Tree-shaking（剔除无用代码），减少打包体积。
- **官方标准**：ECMAScript原生支持，无需额外依赖工具，现代浏览器（Chrome 61+、Firefox 60+等）可直接使用。
- **双向绑定**：导入的模块是只读引用，修改导出模块的内容，导入方会同步更新（与CommonJS不同）。

#### 业务使用场景

所有现代前端项目，包括Vue、React、Angular等框架项目，Vite、Webpack 5+构建的项目，以及需要兼顾浏览器和Node.js的跨端项目（Node14+支持ESM）。

#### 热门程度

目前前端项目使用率95%以上，是前端模块化的“绝对主流”，各类框架、工具库均优先支持ESM，未来会完全替代其他非官方规范。

### 2. CommonJS（CJS）—— Node.js服务端主流

CommonJS是Node.js默认的模块化规范，主要用于服务端开发，语法简洁、易用，是后端模块化的“事实标准”，在前端构建场景中也有少量应用。

#### 核心语法

```javascript
// 1. 导入
const fs = require('fs'); // 内置模块导入
const utils = require('./utils.js'); // 自定义模块导入

// 2. 导出（两种常用方式）
// 方式1：整体导出
module.exports = {
  getName: () => "CommonJS模块",
  age: 20
};

// 方式2：单个导出（挂载到exports）
exports.fn = () => console.log("单个导出");
exports.data = "导出数据";
```

#### 核心特点

- **同步加载**：运行时加载模块，执行到require()时才会加载对应模块，适合服务端（文件读取速度快，不会阻塞）。
- **值拷贝**：导入的是模块导出的“值拷贝”，后续修改导出模块的内容，导入方不会同步更新。
- **Node.js原生支持**：无需额外配置，是Node.js项目的默认选择，生态完善（npm上大部分包均支持CommonJS）。

#### 业务使用场景

Node.js服务端开发（Express、Koa等框架项目）、Webpack构建脚本（配置文件）、本地工具包（如脚本工具、自动化工具），前端项目极少直接使用（会阻塞浏览器渲染）。

#### 热门程度

Node.js服务端使用率100%，是后端模块化的“刚需”，虽然前端已被ESM替代，但由于Node.js的广泛应用，热度始终拉满，与ESM并列最高。

## 四、其他模块化方案简要说明

- **UMD**：兼容CommonJS、AMD和浏览器全局变量，仅用于开发通用SDK、工具库（需适配多环境），非项目开发首选，属于过渡兼容方案。
- **AMD**：异步加载规范，代表工具为RequireJS，仅用于老旧浏览器（IE8及以下）的老项目维护，已完全淘汰，新项目绝对不推荐使用。
- **CMD**：阿里推出的异步规范，代表工具为Sea.js，已完全淘汰，目前几乎无实际应用场景。
- **SystemJS**：浏览器端动态模块加载器，仅用于微前端、低代码平台等小众场景，不是通用模块化方案，使用频率较低。

## 五、核心选型建议

1. **现代前端项目（Vue/React/Vite）**：优先使用ES6 Module，无需考虑其他方案。
2. **Node.js服务端项目**：优先使用CommonJS（默认支持），新项目也可切换为ESM（需配置package.json）。
3. **开发通用工具库/SDK**：可打包为UMD格式（兼容全环境），同时支持ESM和CommonJS。
4. **老项目维护**：AMD/CMD仅用于现有项目兼容，不新增功能；微前端等小众场景可搭配SystemJS。

## 六、总结

前端模块化的核心是 **ES6 Module**（前端首选、官方标准）和 **CommonJS**（后端首选、事实标准），两者热度最高、应用最广，覆盖90%以上的业务场景；其他方案仅用于兼容或小众场景，无需重点掌握，新项目直接聚焦两大核心规范即可。
