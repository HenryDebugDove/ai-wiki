# 核心技术概念汇总（回滚、迁库、集群、主从等）

## 一、基础运维/开发核心概念

| 概念       | 通俗解释                                                                           | 优缺点                                                                     | 实例                                                                               | 常用工具/代码（含Docker实现）                                                                                                                                                                                                                                                         |
| ---------- | ---------------------------------------------------------------------------------- | -------------------------------------------------------------------------- | ---------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 回滚       | 操作出错后，恢复到上一个正常版本/状态（代码、数据库、配置都能回滚）                | 优点：快速恢复业务，操作简单；缺点：无法恢复新产生的数据，分布式系统回滚难 | 1. 新版本上线后用户无法登录，回滚到上一版；2. 误删数据库数据，恢复到备份版本       | 1. Git回滚：git revert HEAD；2. Docker回滚（单机）：停止错误容器→删除→启动旧版本镜像；3. Docker Swarm回滚（推荐）：docker service rollback my-app；4. Docker Compose回滚：修改yml镜像版本，执行docker-compose up -d                                                                 |
| 迁库       | 对数据库做变更，包括改结构（加字段、加索引）、迁数据（换库、搬数据）               | 优点：适配新业务、优化性能；缺点：大表变更会锁表，迁移失败可能丢数据       | 1. 新增“用户积分”功能，给user表加integral字段；2. 订单量暴增，从单库迁到分库分表 | 1. SQL迁库：ALTER TABLE user ADD COLUMN integral INT；2. Docker迁库（核心）：docker exec -i mysql-container mysql -uroot -p密码 库名 < 迁库脚本.sql；3. 临时迁库容器：docker run --rm --link 主库容器:db -v 本地脚本目录:/sql mysql:5.7 mysql -hdb -uroot -p密码 库名 < /sql/脚本.sql |
| 脚本部署   | 用脚本一键完成“拉代码→编译→打包→启动服务”，代替手动操作                       | 优点：自动化、少出错、速度快；缺点：需编写维护脚本，兼容性可能差           | 1. 每日版本更新，运行脚本自动部署；2. 多台服务器批量部署同一服务                   | Docker脚本部署（生产可用）：编写deploy.sh脚本，含git pull、docker build -t 镜像名 .、docker stop/rm 旧容器、docker run 新容器；运行：chmod +x deploy.sh && ./deploy.sh                                                                                                                |
| 不停机更新 | 更新服务时，用户无感知、服务不中断，核心是“先启新版本，再删旧版本”               | 优点：不影响业务、可秒级回滚；缺点：需多实例，新旧版本需兼容               | 1. 外卖平台更新，用户正常下单；2. 抖音更新，用户正常刷视频                         | Docker三种实现：1. Compose（简单）：docker-compose up -d --build --no-deps 服务名；2. Swarm（生产）：docker service update --image 新镜像 服务名；3. 蓝绿部署：先启新版本容器，测试后切换Nginx流量                                                                                    |
| 锁表       | 数据库修改时，把整张表“锁住”，别人暂时不能读写（类似厕所关门，一人用，其他人等） | 优点：保证数据不冲突；缺点：阻塞业务，用户会卡、超时，高峰期易出事故       | 1. 给大订单表加字段，锁表期间用户无法下单；2. 全表更新数据，锁表导致查询超时       | 触发锁表SQL：ALTER TABLE `order` ADD COLUMN `remark` VARCHAR(255)；UPDATE user SET status = 1；Docker环境中，该SQL通过docker exec命令在数据库容器内执行                                                                                                                           |

## 二、数据库架构核心概念（单库、分库、分表）

| 概念 | 通俗解释                                                     | 优缺点                                                                           | 适用场景/实例                                                                                                                                                          | 适用数据库                                                          |
| ---- | ------------------------------------------------------------ | -------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------- |
| 单库 | 所有业务数据（用户、订单、商品）都放在同一个数据库里         | 优点：简单、好维护、联表方便；缺点：数据量大后查询慢，并发扛不住                 | 场景：初创项目、用户万级以内（如小型博客、内部管理后台）；实例：一个MySQL库，里面有user、order、product三张表；Docker环境：启动一个mysql容器，所有服务都连接该容器     | MySQL、PG、SQLite等所有数据库，均可通过Docker容器部署               |
| 分表 | 一张表数据太多，拆成多张结构一样的小表（数据分开，结构相同） | 优点：单表变小、查询变快，避免大表锁表；缺点：跨表查询麻烦，需判断查哪张表       | 场景：单表数据超千万（如订单表、日志表）；实例：order表拆成order_2024、order_2025，按时间拆分；Docker环境：分表逻辑在代码中实现，数据库仍部署在Docker容器内            | MySQL、PG，需代码写路由或用中间件，中间件也可通过Docker部署         |
| 分库 | 把不同业务的数据，放到不同的数据库实例（不同机器）上         | 优点：压力分散、扛高并发，一个库挂了不影响其他；缺点：架构复杂，分布式事务难处理 | 场景：高并发电商、日活百万级；实例：用户服务用user_db，订单服务用order_db，各占一台MySQL；Docker环境：启动多个mysql容器，分别对应不同分库，服务通过容器名/IP连接对应库 | MySQL、PG，需中间件（Sharding-JDBC、MyCat），中间件可通过Docker部署 |

### 分库、分表、单库核心区别

| 维度     | 单库                                    | 分表                                                | 分库                                       |
| -------- | --------------------------------------- | --------------------------------------------------- | ------------------------------------------ |
| 数据位置 | 一个库，多张表（Docker：1个数据库容器） | 一个库，多张结构相同的小表（Docker：1个数据库容器） | 多个库，多台机器（Docker：多个数据库容器） |
| 复杂度   | 低                                      | 中                                                  | 高                                         |
| 并发能力 | 弱                                      | 中                                                  | 强                                         |
| 适用规模 | 小项目                                  | 中大型单表                                          | 高并发大型项目                             |

## 三、服务架构核心概念（分布式服务、集群）

| 概念                 | 通俗解释                                                           | 数据库使用规则（含Docker）                                                                                                                | 实例                                                                                                |
| -------------------- | ------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| 分布式服务（微服务） | 把大系统拆成多个独立小服务（如用户、订单、支付服务），各自独立部署 | 不共用一个库，标准做法是“一个服务一个库”；高并发时，一个服务再分库分表；Docker环境：每个服务部署成独立容器，对应连接一个/多个数据库容器 | 电商系统：用户服务容器→user_db容器，订单服务容器→order_db容器，支付服务容器→pay_db容器，互不干扰 |
| 服务集群             | 同一个服务，启动多个一模一样的实例（目的：扛并发、不停机）         | 99%的情况共用同一个数据库（或同一套主从库）；只有超并发时，才做分库分表；Docker环境：启动多个相同服务容器，均连接同一个/一套数据库容器    | 订单服务集群：3个order-server容器，都连接同一个MySQL容器，用户下单无论进哪个容器，数据都一致        |

## 四、主从数据库（主从复制）

| 核心要点           | 通俗解释/说明（含Docker实现）                                                                                                                                                                                                                                                     |
| ------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 定义               | 1台主库（Master）+ 多台从库（Slave）；主库负责写数据，从库负责读数据，从库自动同步主库数据；Docker环境：主库、从库均部署为独立容器，通过容器链接实现同步                                                                                                                          |
| 优缺点             | 优点：读写分离扛并发、主库挂了可切从库、从库备份不影响主库；缺点：有同步延迟（毫秒~秒级）、架构复杂、成本增加；Docker环境：优点新增“容器部署，环境隔离、启停方便”，缺点不变                                                                                                     |
| 适用场景           | 1. 查询多、写入少（如网站、小程序）；2. 报表统计、数据备份（在从库操作，不卡主库）；3. 主库故障快速切换；Docker环境：适配所有场景，尤其适合快速部署、测试主从架构                                                                                                                 |
| 实例（Docker搭建） | 1. 启动主库：docker run -d --name mysql-master -e MYSQL_ROOT_PASSWORD=123456 mysql:5.7；2. 启动从库（关联主库）：docker run -d --name mysql-slave --link mysql-master:master -e MYSQL_ROOT_PASSWORD=123456 mysql:5.7；3. 进入主库配置同步权限，从库配置同步主库地址，完成自动同步 |
| 代码使用           | 写操作连主库（如创建订单），读操作连从库（如查询订单）；伪代码：createOrder→dbMaster，getOrder→dbSlave；Docker环境：数据库地址填写主从库容器名/IP即可                                                                                                                           |
| 与其他概念关系     | 1. 服务集群 + 主从：多个服务实例共用一套主从库容器；2. 微服务 + 主从：每个微服务有自己的主从库容器，互不干扰                                                                                                                                                                      |

## 五、所有概念关联总结（极简）

| 概念组合（含Docker）               | 常见架构实例                                                                                                                       |
| ---------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| 单库 + 单机服务（Docker）          | 小型博客、内部管理后台（用户少、并发低）；1个服务容器 + 1个数据库容器                                                              |
| 主从库 + 服务集群（Docker）        | 中小型互联网项目（如本地生活小程序）；3个订单服务容器 + 1套主从库容器（1主1从）                                                    |
| 分库分表 + 微服务 + 集群（Docker） | 大型电商（如淘宝、京东）；订单服务集群容器 + 多套order_db分库容器，用户服务集群容器 + 多套user_db分库容器                          |
| 所有概念联动（Docker）             | Docker脚本部署→Docker不停机更新服务集群容器→服务容器操作主从库容器（写主库、读从库）→Docker迁库做结构变更→出错用Docker回滚恢复 |
