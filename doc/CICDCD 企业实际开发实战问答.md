# CI/CD 企业实际开发实战问答（表格版）

## 一、核心基础问答（必问）

| 问题 | 详细答案 | 企业真实业务场景 |
|------|---------|----------------|
| 什么是CI/CD？CI、CD分别是什么？ | 1. **CI（持续集成）**：开发者频繁提交代码到主干，自动触发构建、编译、单元测试，尽早发现代码冲突和Bug；<br>2. **CD（持续交付）**：在CI基础上，自动将代码打包、部署到测试/预发环境，确保代码随时可手动发布；<br>3. **CD（持续部署）**：比持续交付更进一步，无需人工干预，自动将合格代码部署到生产环境。 | 前端团队每天3-5次提交代码，CI自动跑ESLint代码检查、Jest单元测试；CD自动部署到测试环境，产品经理可随时验证新功能，无需开发手动部署。 |
| CI/CD的核心价值是什么？ | 1. 减少人工重复操作，提升发布效率；<br>2. 尽早发现Bug，降低修复成本（开发阶段修复成本远低于生产阶段）；<br>3. 统一开发、测试、生产环境，避免“我本地能跑，服务器跑不起来”；<br>4. 流程标准化、可追溯、可审计，便于团队协作和问题排查；<br>5. 支持快速回滚，降低生产发布风险，提升系统稳定性。 | 某电商项目引入CI/CD后，发布频率从每月1次提升到每天2-3次，线上故障发生率下降70%，故障回滚时间从30分钟缩短至5分钟。 |
| 什么是流水线（Pipeline）？ | Pipeline是将CI/CD全流程（代码拉取→构建→测试→代码质量扫描→制品打包→部署→通知）串联起来的自动化流程，每个环节自动触发，无需人工干预，实现“提交代码即完成部署”的闭环。 | Java微服务项目流水线：Git拉取代码 → Maven编译构建 → JUnit单元测试 → SonarQube代码质量扫描 → 构建Docker镜像 → 推送镜像仓库 → 部署到K8s测试环境 → 接口自动化测试 → 人工审核 → 部署生产。 |
| 企业实际开发中，是不是以Docker+GitLab为主？ | 结论：绝大多数中大型互联网/企业项目，确实以Docker+GitLab为主流，但并非只有这两个工具，而是一套完整工具链。GitLab负责代码托管和CI/CD流水线，Docker负责应用容器化打包，二者是企业CI/CD的核心基础。 | 国内80%以上的中大型公司（如互联网、金融科技、制造业），其微服务项目均采用“GitLab+Docker”为核心，配合Harbor、K8s等工具，实现全流程自动化交付。 |

## 二、Git与分支模型（高频）

| 问题 | 详细答案 | 企业真实业务场景 |
|------|---------|----------------|
| 常用的Git工作流有哪些？ | 1. **GitFlow**：分支体系完整（master/main、develop、feature/*、release/*、hotfix/*），规范严格，适合有固定迭代周期、需求稳定的传统项目；<br>2. **GitHub Flow**：只有main分支+feature分支，feature分支合并到main后直接部署，简单高效，适合云原生、快速迭代的项目；<br>3. **GitLab Flow**：基于多环境分支（dev/staging/prod），兼顾规范与灵活，适合需要多环境验证的企业项目。 | 某银行后端项目用GitFlow，每月1次迭代，严格控制版本和发布节奏；某互联网前端小程序项目用GitHub Flow，每天迭代、随时发布，快速响应用户需求。 |
| Git Hook与CI的区别是什么？ | 1. **Git Hook**：Git自带的本地脚本，在commit、push等事件触发时执行（如pre-commit），常用于本地代码格式化、ESLint检查、简单单元测试；<br>2. **CI**：在服务器端执行，由代码提交推送触发，强制执行所有检查（代码质量、测试、安全扫描），结果可记录、可追溯，不可绕过；<br>**核心区别**：Git Hook在本地，CI在服务器；Git Hook可跳过，CI强制执行。 | 开发者本地commit前，Git Hook自动执行ESLint检查，不合格无法提交；push代码后，GitLab CI再次执行ESLint+单元测试+Sonar扫描，即使本地跳过Hook，服务器也会卡点，防止不合格代码入库。 |

## 三、核心工具相关问答（企业实战重点）

### 3.1 Jenkins相关

| 问题 | 详细答案 | 企业真实业务场景 |
|------|---------|----------------|
| Jenkins流水线有哪两种类型？区别是什么？ | 1. **Freestyle（自由风格项目）**：可视化配置，操作简单，适合简单的构建任务（如单一项目打包），但复杂流程（多环境部署、并行执行）难以维护；<br>2. **Pipeline（Jenkinsfile）**：代码化流水线，支持版本管理、流程复用、并行执行、条件判断，是企业主流选择；<br>Pipeline又分两种：<br>- **Declarative（声明式）**：结构固定、易读，推荐；<br>- **Scripted（脚本式）**：基于Groovy，灵活但复杂。 | 某传统IT公司老项目用Freestyle，仅实现简单的代码打包；微服务新项目统一用Declarative Pipeline，通过Jenkinsfile管理流程，跨项目复用通用逻辑，降低维护成本。 |
| Jenkins如何实现多环境部署（dev/test/prod）？ | 1. 采用参数化构建，让用户选择部署环境（dev/test/prod）；<br>2. 在Jenkinsfile中用when条件判断分支（如main分支部署prod，feature分支部署dev）；<br>3. 不同环境使用不同的配置文件和部署脚本；<br>4. 生产环境增加input人工审核步骤，防止误发布。 | 某电商项目：feature分支提交后，自动部署到dev环境；develop分支合并后，部署到test环境；main分支合并后，触发人工审核，审核通过后部署到prod环境，确保生产发布安全。 |
| Jenkins共享库（Shared Library）是什么？作用是什么？ | Jenkins共享库是将各项目流水线中的通用逻辑（如发送钉钉通知、构建Docker镜像、上传制品到仓库、代码质量扫描）抽成公共的Groovy库，供所有项目复用。<br>**作用**：统一流程规范、减少重复代码、便于维护和升级（修改一处，所有项目生效）。 | 某集团公司有20+微服务项目，将“构建镜像”“发送部署通知”“漏洞扫描”等逻辑封装到共享库，所有项目流水线直接调用，避免每个项目重复编写相同代码，新人上手更快。 |

### 3.2 GitLab CI / GitHub Actions相关

| 工具 | 问题 | 详细答案 | 企业真实业务场景 |
|------|------|---------|----------------|
| GitLab CI | 配置文件名是什么？核心关键词有哪些？ | 配置文件：项目根目录下的`.gitlab-ci.yml`；<br>核心关键词：<br>- `stages`（流水线阶段）<br>- `job`（任务）<br>- `script`（执行脚本）<br>- `only/rules`（触发条件）<br>- `cache`（依赖缓存）<br>- `artifacts`（制品输出） | 前端项目在`.gitlab-ci.yml`中配置cache缓存node_modules，构建时间从3分钟缩短至30秒；仅main分支触发部署任务，其他分支只执行构建和测试。 |
| GitLab CI | 与Jenkins相比，优势是什么？ | 1. 与GitLab代码仓库内置集成，无需单独部署和配置关联；<br>2. 配置简单，通过`.gitlab-ci.yml`即可完成流水线定义，无需复杂的可视化配置；<br>3. 维护成本低，无需单独维护Jenkins服务器和插件；<br>4. 支持内置镜像仓库，可直接存储Docker镜像。 | 某互联网公司放弃Jenkins，改用GitLab CI，减少了服务器维护成本，同时实现代码托管、CI/CD、镜像存储一站式管理，团队协作效率提升40%。 |
| GitHub Actions | 核心概念有哪些？ | 1. **Workflow（工作流）**：整个自动化流程的配置文件，存放在`.github/workflows`目录下；<br>2. **Job（任务）**：工作流中的独立单元，可并行执行；<br>3. **Step（步骤）**：每个Job中的具体操作（如拉代码、安装依赖、构建）；<br>4. **Action（动作）**：第三方预制的可复用步骤（如actions/checkout拉取代码）；<br>5. **Runner（执行器）**：执行工作流的机器（GitHub托管或自建）。 | 某开源项目使用GitHub Actions，通过actions/checkout拉取代码，actions/setup-node安装Node环境，完成构建、测试后，自动发布到npm仓库，全程自动化。 |
| GitHub Actions | 触发方式有哪些？ | 通过`on`关键字配置触发方式，常见：<br>1. `push`：代码推送到指定分支触发；<br>2. `pull_request`：提交PR时触发；<br>3. `release`：创建发布标签时触发；<br>4. `schedule`：定时触发（如每天凌晨跑回归测试）。 | 某云原生项目配置：push到main分支触发CI构建，创建release标签触发生产部署，每天凌晨2点定时执行全量回归测试，确保系统稳定性。 |

### 3.3 Docker + K8s（云原生CI/CD必问）

| 问题 | 详细答案 | 企业真实业务场景 |
|------|---------|----------------|
| CI/CD中Docker的作用是什么？典型命令有哪些？ | **作用**：<br>1. 统一环境：开发、测试、生产使用同一个Docker镜像，彻底解决环境依赖冲突；<br>2. 隔离依赖：不同应用的依赖相互隔离，避免干扰；<br>3. 标准化交付：镜像作为统一的交付物，可直接部署，无需重新配置环境；<br>4. 版本管理：镜像可打版本标签，便于追溯和回滚；<br>**典型命令**：<br>- `docker build -t 镜像名:版本号 .`（构建镜像）<br>- `docker push 镜像名:版本号`（推送镜像到仓库） | 某微服务项目，Java、Node、Python三种语言的应用，均通过Docker打包，镜像推送到Harbor仓库，测试、生产环境直接拉取镜像部署，避免了“本地能跑，线上不行”的问题；每次提交代码，镜像标签用commit id命名，便于追踪代码版本。 |
| CI/CD结合K8s怎么做？如何实现发布和回滚？ | **CI/CD+K8s流程**：<br>1. 代码提交触发CI流水线，构建Docker镜像并推送至镜像仓库；<br>2. 流水线调用kubectl命令，更新K8s的Deployment资源，指定新的镜像版本；<br>3. K8s采用滚动更新策略，逐步替换旧容器，确保服务不中断；<br>**发布命令**：<br>`kubectl set image deployment应用名 容器名=镜像名:版本号 -n 命名空间`<br>**回滚命令**：<br>`kubectl rollout undo deployment应用名 -n 命名空间` | 某电商核心服务部署在K8s集群，每次发布通过CI/CD流水线自动更新镜像，滚动更新过程中，服务可用率保持100%；一次发布出现Bug，执行回滚命令，10秒内恢复到上一版本，未影响用户使用。 |
| 什么是蓝绿发布、金丝雀发布？企业如何选择？                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               | 1. 蓝绿发布：部署两套完全相同的环境（蓝环境、绿环境），一套运行当前版本，一套部署新版本；验证通过后，切换流量到新版本，回滚时直接切换回旧环境，无停机风险；2. 金丝雀发布：先将新版本部署到少量服务器（金丝雀节点），只分配少量流量验证；验证通过后，逐步扩大部署范围，直至全量发布；选择原则：核心关键服务（如支付、订单）用金丝雀发布，降低风险；非核心服务（如后台管理）用蓝绿发布，简化流程。 | 某支付平台的支付接口用金丝雀发布，先给1%用户分配新版本流量，观察1小时无异常后，逐步扩大到10%、50%、100%；后台管理系统用蓝绿发布，部署新版本后直接切换流量，快速完成发布。                                             |

| 企业如何管理制品（Jar、Docker镜像）？                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   | 1. 制品仓库选择：Jar包用Nexus，Docker镜像用Harbor/GitLab内置镜像仓库；2. 版本管理：镜像/Jar包打上清晰的版本标签（如commit id、时间戳、版本号），便于追溯；3. 清理策略：保留最近10-20个版本，定期清理过期制品，节省存储空间；4. 权限控制：给不同团队分配不同的制品仓库权限，防止误操作。                                                                                                              | 某企业用Nexus存储Java Jar包，Harbor存储Docker镜像，每个镜像都打上“项目名:commit id”标签，保留最近15个版本，出问题时可快速回滚到历史版本，同时定期清理3个月前的过期镜像，节省服务器空间。                                                                                                                                                                                                                       |                                                                                                                                                                                                     |
| 安全规范                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                | SAST、SCA、DAST的区别是什么？企业如何应用？                                                                                                                                                                                                                                                                                                                                                          | 1. SAST（静态应用安全测试）：在代码编译前扫描，检测代码本身的安全漏洞（如SQL注入、空指针），工具：SonarQube、Checkmarx；2. SCA（软件成分分析）：扫描第三方依赖包的漏洞，工具：Dependency-Check、Nexus IQ；3. DAST（动态应用安全测试）：在服务运行时扫描，检测运行时漏洞（如跨站脚本），工具：OWASP ZAP；企业应用：CI阶段加入SAST+SCA卡点，不通过则无法进入下一阶段；DAST放入测试环境部署后执行，确保运行时安全。 | 某金融公司，在CI流水线中加入SonarQube（SAST）和Dependency-Check（SCA），代码漏洞未修复、依赖漏洞未处理的情况下，无法打包镜像；测试环境部署后，自动执行OWASP ZAP扫描，发现运行时漏洞后通知开发修复。 |
| 企业如何防止CI/CD中的敏感信息泄露？                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     | 1. 不硬编码敏感信息：禁止在代码、配置文件中硬编码密码、Token、密钥；2. 使用平台凭据管理：通过Jenkins Credentials、GitLab CI/CD Variables存储敏感信息，在流水线中引用；3. 密钥管理服务：大型企业使用Vault等工具管理密钥，动态获取，避免静态存储；4. 镜像日志脱敏：Docker镜像不打包敏感信息，构建/部署日志脱敏（隐藏密码、Token）；5. 权限控制：严格控制CICD平台、制品仓库的访问权限，避免未授权访问。 | 某互联网公司，数据库密码、API Token存储在GitLab CI/CD Variables中，流水线中通过变量引用，不直接写在脚本中；日志中自动隐藏敏感信息，同时只有开发、运维人员有权限访问CI/CD平台和Harbor仓库，防止敏感信息泄露。                                                                                                                                                                                                     |                                                                                                                                                                                                     |
